# 七點目標驗收標準

本檔把使用者 `/goal` 的七點要求轉成可驗收的專案標準。任何交付都必須同時滿足以下七點，不可只完成其中一部分就宣稱完成。

## GOAL-1 真實上線內部遊玩測試

- 後端必須部署到線上內測 API。
- Godot 客戶端必須可連線上 API。
- Windows 內測版必須可直接啟動。
- 外部金流可暫緩，但內部經濟、模擬付款、禮包、角色、戰鬥、背包、技能與狀態保存必須跑通。

## GOAL-2 每個環節需驗證方向

- 每次核心變更必須跑自動檢查。
- 不可用與專案方向無關的玩法或 UI 補洞。
- 更好的建議只能列為後續調整，不可直接替換使用者指定方向。

## GOAL-3 原創 RO-like 美術定位

- 風格只能是原創、明亮、2D/2.5D、RO-like 類型定位，不可複製 RO 素材。
- 必須有角色、怪物、物品、技能、禮包、場景等基本美術或 Godot 繪製圖示。
- 必須有男女基礎紙娃娃。
- 裝備必須能在角色身上顯示武器、衣服、頭飾、飾品差異。

## GOAL-4 全程使用 SKILL

- Godot 開發使用 `godot`、`godot-ui`、`hud-system`、`2d-essentials`、`animation-system`、`inventory-system`、`gdscript-patterns`。
- 後端/API 使用 `senior-backend`。
- 驗證與內測門檻使用 `senior-qa`。
- 使用紀錄必須保存在 `docs/implementation-skill-usage.md`。

## GOAL-5 遊戲角度與 UX

- 玩家端必須是遊戲介面，不可呈現工程後台、營運、稅務、提現、風控管理文案。
- 文字主要用於戰鬥、掉寶、探索、角色目前執行狀態。
- 狀態資訊與可按功能必須視覺區隔。

## GOAL-6 首頁內容邊界

首頁只允許呈現：

- 角色與紙娃娃外觀。
- 角色資訊。
- 地區場景。
- 角色目前執行狀態文字欄。
- 功能選單。

首頁不得呈現禮包持有、禮包交易進度、交易場細節、營運結算或提現資訊；這些必須放在對應功能頁。

## GOAL-7 開始前不足處先確認

使用者已補充並確認：

- 原創為主，只參考 RO 類型定位。
- 目標是部署上線給少數人測試。
- 男女角色各一套基礎紙娃娃。
- 外部金流暫緩，內部經濟與模擬付款必須跑通。
- 必須使用 Godot，不使用 CSS/網頁方式製作玩家端美術。

因此目前不再以「需求不明」作為停止理由，除非遇到真實憑證、付費平台、簽章、帳號或第三方授權阻塞。
