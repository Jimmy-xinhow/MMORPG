# 01 Product Boundary and Withdrawal Policy

## Purpose

This document defines what the product is allowed to do, what must never be withdrawable, and how operating income can be withdrawn in a legal, taxable, auditable way.

The system is an MMORPG-style lucky pack resource game. It may contain virtual packs, in-game currency, resource items, tickets, boss rewards, guild activity, and operator income programs. It must not present ordinary gameplay as a cash earning opportunity.

## 中文執行摘要：不可提現與可提現邊界

本專案的基準規則是：玩家遊戲資產不可提現，經營者收入才可能在合法合規、完成稅務資料、風控審核、會計審核與付款流程後提現或撥款。

明確不可提現的內容：

- GC、遊戲幣、贈送幣、活動幣不可提現。
- 幸運禮包本身不可提現。
- 禮包開出的道具、資源、裝備、碎片、Boss 材料不可提現。
- 票券、Boss 票、公會票、商城票、活動票不可提現。
- Boss 獎勵、公會獎勵、排行榜獎勵、賽季獎勵不可提現。
- 玩家交易產生的價格差、轉售差、帳面增值不可提現。
- 平台補償、活動贈品、測試資產、客服手動補發資產不可提現。
- 退款、爭議款、退單、風控凍結、疑似洗交易、同設備/同銀行帳戶群組相關金額不可提現。

可提現或可撥款的內容只限於經營者計畫中的 `OperatorSettlement`：

- 必須是經營服務、活動營運、內容品質、客服品質等經核准 KPI 或合約服務產生的經營收入。
- 必須排除玩家遊戲資產、玩家交易利益、票券、Boss 獎勵、促銷資產與任何風控高風險來源。
- 必須通過身分/稅務/付款資料檢查，並保留報稅與會計稽核資料。
- 必須能輸出結算期間、核准金額、排除金額、稅務狀態、審核人與付款紀錄。

## Core Product Boundary

- Player gameplay rewards are entertainment assets only.
- Lucky pack contents are consumed inside the game economy unless a later regulated review explicitly approves another treatment.
- Player-to-player pack trading is a game-market mechanic, not a cash-out channel.
- Operator income is a separate business settlement program tied to approved operating contributions, invoices/tax records, risk review, and payout workflow.
- Any screen, copy, API, report, or export that may look like "cash value" must name the asset type clearly.

## Non-Withdrawable Content

The following content is never withdrawable by default:

| Content | Examples | Reason |
|---|---|---|
| Game coin | GC, bonus coin, promo coin | Internal game balance only |
| Lucky pack | unopened pack, listed pack, traded pack | Random virtual item container |
| Pack content | item, resource, equipment, fragment, boss material | Gameplay asset only |
| Ticket | boss ticket, shop ticket, event ticket | Access/right-to-play asset, not money |
| Boss reward | boss drop, contribution reward, guild reward | Gameplay reward |
| Ranking reward | leaderboard points, season title, medal | Non-cash recognition unless reviewed |
| Promotional grant | coupon, free pack, airdrop, compensation item | Marketing/ops grant |
| Locked reward | risk-held reward, suspicious reward, disputed reward | Not eligible before review |
| Test or admin asset | seeded asset, QA grant, support adjustment | No economic entitlement |
| Refunded or reversed value | chargeback-related value, cancelled transaction value | Must not be re-withdrawn |

The UI and API must not use words such as `可提現`, `收入`, `現金`, `收益`, or `分潤` for the items above. Use `遊戲資產`, `遊戲道具`, `票券`, `資源`, or `活動獎勵`.

## Withdrawable Operating Income

Only approved operating income can become withdrawable. It must be generated by an `OperatorSettlement` record and must pass tax, risk, and payment checks.

Allowed sources:

| Source | Required Evidence | Notes |
|---|---|---|
| Approved operator KPI settlement | `OperatorKpiSnapshot`, settlement period, approval log | Calculated from approved business contribution only |
| Contracted campaign service fee | campaign contract/ref, delivery proof, invoice/tax form | Must be outside ordinary player reward loop |
| Approved platform service commission | commission policy version, transaction evidence, accounting review | Must not be represented as player cash-out |
| Manual accounting adjustment | adjustment reason, approver, audit note | Requires finance admin role and immutable log |

Disallowed operating income sources:

- Direct conversion from GC, packs, tickets, boss drops, or gameplay rewards.
- Income derived from same-device, same-bank, wash-trade, self-dealing, or suspicious clusters.
- Income linked to chargeback, refund, dispute, pending fulfillment, or policy violation.
- Income for users without completed identity, tax, and payment information.
- Income created without an immutable settlement period and approval record.

## System-Level Conversion Matrix

| From | To | Allowed? | Enforcement |
|---|---:|---:|---|
| GC | Cash withdrawal | No | API reject, no ledger route |
| Pack | Cash withdrawal | No | API reject, UI hidden |
| Pack content | Cash withdrawal | No | API reject, asset type guard |
| Ticket | Cash withdrawal | No | API reject, liability ledger only |
| Boss reward | Cash withdrawal | No | API reject, reward ledger only |
| Player trade profit | Cash withdrawal | No | no `OperatorSettlement` source |
| Operator settlement | Withdrawal request | Yes, after review | settlement status and risk checks |
| Approved withdrawal | Payment batch | Yes | tax-ready and payment approval |

No database migration, service, cron job, or admin action may add a direct route from non-withdrawable assets to a withdrawal account.

## Exclusion Priority

When calculating withdrawable operating income, exclusions are applied in this order:

1. Legal or policy block: prohibited jurisdiction, missing tax profile, banned account.
2. Payment integrity block: chargeback, refund, dispute, duplicate payout, payment failure.
3. Identity block: failed KYC/KYB, unverified bank account, expired tax document.
4. Abuse block: same device, same payment account, self-dealing, wash cluster, bot signal.
5. Settlement block: unapproved settlement, unlocked period, stale config version.
6. Manual finance hold: accounting discrepancy, support case, regulator request.

If any exclusion applies, `withdrawableAmount` must be `0` for the affected settlement portion until the block is resolved.

## Withdrawal State Machine

```text
DRAFT
-> PENDING_REVIEW
-> APPROVED
-> TAX_READY
-> PAYMENT_SCHEDULED
-> PAID
```

Failure or stop states:

```text
PENDING_REVIEW -> REJECTED
APPROVED -> FROZEN
TAX_READY -> CANCELLED
PAYMENT_SCHEDULED -> PAYMENT_FAILED -> PENDING_REVIEW
```

Required guards:

- `DRAFT -> PENDING_REVIEW`: settlement exists, user owns operator account, amount > 0.
- `PENDING_REVIEW -> APPROVED`: risk reviewer approves, no open high-risk case.
- `APPROVED -> TAX_READY`: tax profile valid, payout identity valid.
- `TAX_READY -> PAYMENT_SCHEDULED`: finance approval and payment batch created.
- `PAYMENT_SCHEDULED -> PAID`: bank transfer confirmation or payment provider success.
- Any state -> `FROZEN`: risk, legal, chargeback, or compliance hold.

## UI Disclosure Requirements

The product must disclose these boundaries in:

- Pack purchase screen.
- Pack opening screen.
- Trading/listing screen.
- Ticket shop.
- Boss and guild reward screen.
- Operator center.
- Withdrawal request screen.
- Terms, FAQ, and account risk notice.

Required wording principle:

```text
遊戲資產不可提現。只有經營者計畫中已核准、完成稅務與風控審查的經營收入，才可申請撥款。
```

## Acceptance Tests

- A user cannot create a withdrawal request from GC, pack, ticket, boss reward, or trade profit.
- A withdrawal request without `OperatorSettlement` is rejected by backend validation.
- A risk-held settlement cannot move to `TAX_READY`.
- A settlement with expired tax profile cannot move to `PAYMENT_SCHEDULED`.
- Product copy scan finds no player-facing text implying ordinary gameplay cash-out.
- Finance export includes taxable amount, withholding fields if applicable, settlement period, and approval trail.
